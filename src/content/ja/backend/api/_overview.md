# API 設計知識体系

## 核心概念

API はシステム間通信のコントラクトで、良い API 設計はシステムの使いやすさと保守性を決定します。

### API スタイル比較

```
三つの主流 API スタイル
┌─────────────────────────────────────────────┐
│           REST                              │
│  • リソース指向、HTTP セマンティクス          │
│  • シンプル直感的、広くサポート              │
│  • CRUD 操作に適している                    │
├─────────────────────────────────────────────┤
│           GraphQL                           │
│  • クエリ言語、必要なデータのみ取得          │
│  • 単一エンドポイント、リクエスト削減        │
│  • 複雑なデータ関係に適している              │
├─────────────────────────────────────────────┤
│           gRPC                              │
│  • Protocol Buffers、バイナリ               │
│  • 双方向ストリーム、高性能                  │
│  • マイクロサービス内部通信に適している      │
└─────────────────────────────────────────────┘
```

## 学習パス

### RESTful API

- **リソース設計** - URI 規範、命名規則
- **HTTP メソッド** - GET, POST, PUT, DELETE
- **ステータスコード** - 2xx, 4xx, 5xx セマンティクス
- **バージョン管理** - URL バージョン、Header バージョン

### GraphQL

- **Schema 定義** - Type, Query, Mutation
- **クエリ構文** - フィールド選択、ネストクエリ
- **リゾルバー** - Resolver 実装
- **最適化** - DataLoader, キャッシュ

### gRPC

- **Protobuf** - メッセージ定義
- **サービス定義** - RPC メソッド
- **ストリーム通信** - 単方向、双方向ストリーム
- **インターセプター** - ミドルウェアパターン

## ベストプラクティス

1. **一貫性** - 統一された命名とレスポンス形式
2. **ドキュメント化** - OpenAPI/Swagger 規範
3. **バージョン管理** - 互換性と進化
4. **エラーハンドリング** - 明確なエラーメッセージ
